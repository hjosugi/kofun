# HTTP client contract

Status: accepted v1 design.

Issue: [#638](https://github.com/hjosugi/kofun/issues/638).

Tier: official independently versioned module.

## Outcome

Kofun provides one first-party HTTP client path for ordinary API calls without
hiding network authority, clocks, entropy, root trust, proxy configuration,
redirects, retries, buffering, or resource limits.

The first implementation is an HTTP/1.1 protocol core over an injected
byte-stream transport. Plain loopback sockets are the first live adapter.
HTTPS is part of the public contract, but it returns an explicit unsupported
provider error until an audited TLS adapter is selected. HTTP/2, HTTP/3, and
WebSocket are later protocol modules.

HTTP semantics follow RFC 9110 and HTTP/1.1 parsing/framing follows RFC 9112.
URI syntax follows RFC 3986. The implementation must not recover from invalid
framing in a way that could make two participants disagree about message
boundaries.

## Module boundary

The conceptual public modules are:

```text
official.net.url
official.net.http
official.net.http.transport
official.net.tls
```

`official.net.http` owns public requests, responses, redirect policy, body
limits, cancellation, and typed errors. `transport` is a narrow injected
connection interface. URL and TLS may be released independently, but a client
release pins the compatible major versions.

The names are conceptual until the package manifest and naming work assigns
the final first-party package identity. They do not occupy `std` while the
module remains Tier 3.

## Capabilities

Constructing a client requires explicit capabilities:

```text
Client.new(
    resolver,
    connector,
    clock,
    tls_provider,
    trust_store,
    configuration,
) -> Result[Client, ClientConfigurationError]
```

- `resolver` resolves names; it does not read ambient resolver configuration
  unless the application selected a system resolver adapter.
- `connector` opens byte streams and owns the network effect.
- `clock` supplies monotonic deadlines. Tests inject a fake clock.
- `tls_provider` is explicit. A plaintext-only provider rejects `https`.
- `trust_store` is explicit. A system-root adapter may be selected by the
  application, but the library does not discover filesystem roots by itself.
- proxy and credential providers are optional explicit configuration.

The client never reads proxy, credential, certificate, locale, or timeout
settings from the environment implicitly.

## URL profile

V1 accepts absolute `http` and `https` URLs.

```text
Url.parse(text: Text) -> Result[Url, UrlError]
```

Rules:

- the scheme and host are required;
- fragments are rejected for requests;
- user-info is rejected; credentials belong in explicit headers/providers;
- an empty path is normalized to `/`;
- the scheme and ASCII host are case-normalized;
- the port is checked in `1..65535`; the default port is omitted from canonical
  display;
- percent escapes are exactly two hexadecimal digits and are not decoded and
  re-encoded opportunistically;
- dot-segment removal follows RFC 3986 for resolution;
- only ASCII host names and already encoded `xn--` labels are accepted in v1;
  Unicode IDNA conversion is a separate reviewed module;
- IPv4 and bracketed IPv6 literals are represented distinctly from DNS names;
- control characters, spaces, invalid UTF-8, forbidden delimiters, and
  ambiguous authority forms fail.

`Url.resolve(base, reference)` implements RFC 3986 reference resolution. It
does not perform I/O.

## Request

```text
Request {
    method: Method,
    url: Url,
    headers: HeaderMap,
    body: RequestBody,
}
```

`Method` validates the RFC token grammar and preserves the canonical spelling
provided by a typed known method or an explicitly constructed extension
method.

Headers:

- names are validated tokens and compared case-insensitively;
- values reject CR, LF, NUL, and other forbidden controls;
- connection-specific headers are owned by the HTTP/1.1 adapter;
- `Host`, framing headers, and proxy authorization cannot conflict with values
  generated by the client;
- sensitive headers are marked so redirect policy can remove them.

`RequestBody` is:

```text
Empty
Bytes(read Bytes)
Text(read Text, content_type: Text)
Stream(take BodySource, known_length: Optional[Int])
```

A stream is affine and can be consumed once. It is replayable only when the
source explicitly implements `ReplayableBody`. Unknown-length HTTP/1.1 bodies
use chunked transfer coding. Content length is checked and never inferred by
reading an unbounded stream into memory.

Convenience JSON requests encode into a configured maximum byte buffer and
return `BodyLimitExceeded` before network I/O if that maximum is exceeded.

## Response and body ownership

```text
Response {
    status: Status,
    headers: HeaderMap,
    body: take ResponseBody,
}
```

`ResponseBody` owns the connection lease. It supports bounded reads and an
explicit stream:

```text
read_chunk(edit body, max_bytes: Int) -> Result[Optional[Bytes], HttpError]
read_all(take body, limit: Int) -> Result[Bytes, HttpError]
close(take body) -> Result[Void, HttpError]
```

Normal end-of-body returns the connection to the pool only after framing is
complete. `close` may drain at most `configuration.max_drain_bytes`; if more
bytes remain, framing is uncertain, cancellation occurred, or an error was
seen, it closes the transport instead of reusing it.

Dropping a body invokes the same bounded close policy. It never spawns
unbounded background draining. A body or connection cannot be used after
`take`.

## Normative use examples

The syntax below is conceptual until the module/package naming work lands. The
ownership and failure paths are normative.

### Bounded GET

```text
let request = Request.get(Url.parse("http://api.example.test/items?limit=2")?)
let response = client.send(take request, cancel, deadline)?
let bytes = read_all(take response.body, limit: 64 KiB)?
```

`send` does not consult ambient proxy, credential, clock, DNS, or trust state.
The injected capabilities are the only authority. `read_all` either returns a
complete body within 64 KiB or a typed error; it never returns a successful
prefix.

### Bounded JSON POST

```text
let body = JsonBody.encode(read value, limit: 256 KiB)?
let request = Request.post(url, Body.Bytes(read body))
    .header("content-type", "application/json")
let response = client.send(take request, cancel, deadline)?
```

Encoding and the 256 KiB limit are checked before network I/O. The request is
replayable because it owns bounded bytes, but v1 still performs no automatic
retry. A 301, 302, 307, or 308 response does not rewrite or automatically
resend this `POST`.

### Streamed response

```text
let response = client.send(take request, cancel, deadline)?
let body = take response.body
let total = 0

while let Some(chunk) = read_chunk(edit body, max_bytes: 16 KiB)? {
    total = checked_add(total, chunk.len())?
    require(total <= 8 MiB, BodyLimitExceeded)
    consume(read chunk)?
}

close(take body)?
```

Each read is bounded by both the call limit and the client's total deadline.
End-of-body makes the connection reusable only after valid framing completes.
Cancellation, truncation, invalid framing, or a total limit failure closes the
transport.

### Early body close

```text
let response = client.send(take request, cancel, deadline)?
close(take response.body)?
```

`close` drains no more than `max_drain_bytes`. It returns a reusable connection
only when the remaining body and framing complete within that bound; otherwise
it closes the transport. Dropping the body has exactly this policy.

### Redirect and cancellation outcomes

| event | required outcome |
|---|---|
| same-origin GET receives 302 within redirect budget | follow using GET |
| cross-origin GET receives 302 | follow only under explicit policy and strip sensitive headers |
| POST receives 302 | return the redirect response; do not resend |
| redirect would change HTTPS to HTTP | reject unless explicitly unsafe policy was selected |
| cancellation before/during connect, write, headers, or body | return `Cancelled` and do not reuse uncertain transport |
| total deadline expires in any phase | return `Timeout` and do not begin another redirect |

## Limits

Every client has finite positive limits:

```text
max_connections_total
max_connections_per_origin
max_idle_connections
max_header_bytes
max_header_fields
max_status_line_bytes
max_body_bytes
max_decoded_body_bytes
max_chunk_line_bytes
max_trailer_bytes
max_redirects
max_drain_bytes
connect_timeout
header_timeout
body_idle_timeout
total_timeout
```

The default profile is:

```text
connections total/per-origin: 64 / 8
idle connections: 16
header bytes/fields: 32 KiB / 100
status line: 8 KiB
body bytes: 16 MiB
decoded body bytes: 32 MiB
chunk line: 8 KiB
trailers: included in the header byte/field budgets
redirects: 5
drain bytes: 64 KiB
connect/header/body-idle/total timeout: 10 s / 10 s / 30 s / 60 s
```

Callers may lower or explicitly raise these values with checked configuration.
Zero, negative, overflowed, or internally inconsistent limits fail client
construction.

## Framing and decompression

V1 supports HTTP/1.1 response bodies framed by:

- no-body response semantics;
- one valid `Content-Length`;
- valid chunked transfer coding;
- connection close only where RFC 9112 permits it.

Conflicting or malformed content lengths, unsupported transfer codings,
invalid chunks, forbidden trailer fields, premature EOF, excessive interim
responses, and ambiguous framing close the connection and return a protocol
error.

Automatic decompression is disabled by default. When enabled for a supported
encoding, both encoded and decoded byte limits apply. Hitting either limit
closes the response body. The client never returns a successful truncated
decoded value.

## Redirects

The default is:

```text
RedirectPolicy.same_origin_safe(max_hops: 5)
```

Rules:

- only `GET` and `HEAD` are followed automatically;
- 301, 302, 303, 307, and 308 are recognized;
- 303 changes `HEAD` to `HEAD` and other followed methods to `GET`;
- 307/308 preserve method/body only when the body is empty or replayable;
- cross-origin redirects require an explicit policy;
- authorization, cookie, proxy authorization, and caller-marked sensitive
  headers are removed on cross-origin redirects;
- HTTPS-to-HTTP downgrade is rejected unless a dedicated unsafe policy is
  selected;
- relative `Location` uses `Url.resolve`;
- missing/invalid `Location`, loops, and the hop limit are typed errors;
- each intermediate body follows the bounded close/drain rule.

No redirect occurs after the total deadline.

## Retries

V1 performs no automatic application request retry.

A failure exposes a conservative `retry_class`:

```text
Never
SafeIfIdempotentAndReplayable
AfterExplicitPolicy
```

The classification is advice, not an action. The library cannot know whether a
server processed a request before a connection failed. Higher-level retry code
must select an idempotency and backoff policy explicitly.

## Cancellation and timeouts

Every send accepts an explicit cancellation token and optional deadline:

```text
send(edit client, take request, cancel, deadline) ->
    Result[Response, HttpError]
```

The effective deadline is the earliest of the request deadline and configured
phase/total timeouts. Cancellation is checked during resolution, connection,
TLS handshake, writes, header reads, body reads, redirects, and bounded drain.

Cancellation closes a transport unless the adapter proves the canceled
operation left framing reusable. No detached task continues after `send` or
body close returns.

## TLS

HTTPS requires a Tier 3 TLS provider with:

- TLS 1.2 or later by default;
- hostname verification against the URL host;
- SNI for DNS names;
- certificate-chain verification to the selected trust store;
- secure entropy from the explicit entropy capability;
- certificate, handshake, record, and total deadline limits;
- no trust-all or hostname-disable option in the safe API;
- provider/root-store version identities in diagnostics and reproducibility
  metadata;
- an update channel independent from compiler releases.

Unsupported algorithms, protocol versions, trust stores, and targets fail
explicitly. Implementing cryptographic primitives in the HTTP module is a
non-goal.

## Errors

```text
HttpError =
    InvalidUrl(UrlError)
  | Resolve(ResolveError)
  | Connect(ConnectError)
  | Tls(TlsError)
  | Timeout(phase)
  | Cancelled(phase)
  | RequestBody(RequestBodyError)
  | Protocol(ProtocolError)
  | HeaderLimitExceeded
  | BodyLimitExceeded(encoded, decoded)
  | Redirect(RedirectError)
  | UnsupportedProvider(feature)
```

Errors record the operation phase and stable code. They do not include
credentials, authorization/cookie values, full request bodies, or private key
material. Partial response data is not returned as success.

## Pooling

The pool is per client and keyed by scheme, canonical host, port, proxy
identity, TLS provider/trust identity, and protocol parameters. A connection is
never reused across keys.

Idle eviction is driven by the injected monotonic clock. Limits use deterministic
least-recently-returned order with a stable tie break. Client close takes the
pool and closes every connection; it does not wait on detached requests.

## Deterministic test transport

The protocol core must run over a scripted in-memory transport that can:

- fragment every byte boundary;
- delay each operation through a fake clock;
- inject EOF, cancellation, and transport failures;
- record exact request bytes;
- serve pipelined/interim/chunked/trailer responses;
- prove connection reuse or closure.

The live acceptance transport uses an ephemeral loopback server only. Tests do
not depend on public DNS, public CAs, the Internet, the host proxy environment,
or the wall clock.

Required negative fixtures include request/response splitting, conflicting
lengths, transfer-coding ambiguity, invalid chunks, oversized fields, slow
headers/body, decompression expansion, redirect loops/downgrades, sensitive
header forwarding, cancellation in every phase, and early body drop.

## Implementation slices

1. URL value and deterministic resolution.
2. HTTP/1.1 request/response core over the scripted transport.
3. Plain loopback socket adapter and bounded pool.
4. Redirect/cancellation integration.
5. Audited TLS provider adapter and HTTPS acceptance.
6. Optional decompression providers.

Only slice 2 is the first HTTP client implementation issue. It consumes the URL
contract but does not wait for TLS.
